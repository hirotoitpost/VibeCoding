# 📚 ドキュメント管理スキル（VibeCoding プロジェクト）

**このスキルはドキュメント作成・修正・再編成の判断フロー**を提供します。  
🔧 **Git コミット・PR 規約は別スキルを参照してください** → [Git 規約スキル](.instructions.md)

---

## 🚀 使用シーン別ガイド

### シーン 1: 新規ドキュメントを作成する時

**質問**: 「新しく `新規ファイル.md` を作成したい」  
**参照フロー**:

```
1️⃣ DOCUMENTATION_CREATION_CHECKLIST.md
   ↓
   「このドキュメントは何 Tier？」
   
2️⃣ DOCUMENTATION_STRATEGY.md で Tier 確認
   ↓
   Tier 1-7 のどれか？ → 行数目安決定
   
3️⃣ 参照関係を整理
   ↓
   「既存ドキュメントとどこで関連する？」
   
4️⃣ スキル化判定
   ↓
   「独立した SKILL として切り出すべき？」
   
5️⃣ 作成実施
```

**参考リンク**:
- [DOCUMENTATION_CREATION_CHECKLIST.md](DOCUMENTATION_CREATION_CHECKLIST.md) - 作成時確認フロー
- [DOCUMENTATION_STRATEGY.md](DOCUMENTATION_STRATEGY.md#-ドキュメント体系) - Tier 定義

**具体例**:
```
Q: ID 028 向けにプロジェクト README を作成します
A: 
  1. Tier 6（プロジェクト別）とわかった
  2. 行数目安: 100-150行
  3. 既存: examples/新規フォルダ/README.md で参照
  4. SKILL化: 不要（プロジェクト専有）
  5. 作成実施 ✅
```

### シーン 2: ドキュメントが肥大化した時

**質問**: 「APP_CANDIDATES.md が 500行を超えた」  
**参照フロー**:

```
1️⃣ DOCUMENTATION_STRATEGY.md を確認
   ↓
   目安: Tier 3 は 200-300行
   
2️⃣ 分割戦略を立案
   ↓
   「どのセクションを分割するか？」
   → APP_CANDIDATES_ADVANCED.md への抽出
   → APP_CANDIDATES_OVERVIEW.md の整理
   
3️⃣ クロスリファレンス設計
   ↓
   「どこから参照するか？」
   → 親ファイルに目次＆リンク追加
   
4️⃣ 実施＆検証
   ↓
   各ファイル行数確認 ✅
```

**参考リンク**:
- [DOCUMENTATION_STRATEGY.md](DOCUMENTATION_STRATEGY.md#-定期リファクタリング計画) - リファクタリング計画
- [APP_CANDIDATES.md](APP_CANDIDATES.md) - 実装例（503行→172行+4分割）

**具体例**:
```
Q: GIT_WORKFLOW.md が 300行を超えた
A:
  1. 目安確認: Tier 2 は 150-200行（超過中）
  2. 分割: GIT_COMMIT_CONVENTION.md / GIT_BRANCH_STRATEGY.md へ抽出
  3. 参照: GIT_WORKFLOW.md に「参照先」リンク追加
  4. 検証: 各ファイル行数確認＆参照リンク動作確認
```

### シーン 3: ドキュメント参照関係を整理する時

**質問**: 「agents.md が複数の場所から参照されている」  
**参照フロー**:

```
1️⃣ 参照元を把握
   ↓
   grep, semantic_search で参照箇所を列挙
   
2️⃣ 最上位ドキュメント＆ツリーを整理
   ↓
   参照の階層化（親→子→孫）
   
3️⃣ DOCUMENTATION_STRATEGY.md で Tier を確認
   ↓
   Tier 1 が最上位？
   
4️⃣ クロスリファレンス更新
   ↓
   各 Tier でのリンク方式統一
   
5️⃣ 検証＆ドキュメント化
```

**参考リンク**:
- [DOCUMENTATION_STRATEGY.md](DOCUMENTATION_STRATEGY.md#-ドキュメント体系) - Tier 体系図
- [agents.md](agents.md) - 参照関係の実装例

---

### シーン 4: セッション終了時（ドキュメント監視）

**質問**: 「セッション完了後、ドキュメント全体の状態を確認したい」  
**実施フロー**:

```
1️⃣ 全ドキュメント行数を統計
   ↓
   Get-ChildItem *.md | 行数集計
   
2️⃣ DOCUMENTATION_STRATEGY.md と比較
   ↓
   各 Tier の目安に収まっているか？
   
3️⃣ 警告基準をチェック
   ↓
   👁️ 200行超過: 注視対象
   🚨 400行超過: 即座に分割検討
   
4️⃣ SESSION_PROGRESS_EXTENDED.md に記録
   ↓
   セッション統計に行数を追加
   
5️⃣ リファクタリング計画の立案
   ↓ 必要に応じて次セッションアクション化
```

**参考リンク**:
- [DOCUMENTATION_STRATEGY.md](DOCUMENTATION_STRATEGY.md#-定期リファクタリング計画) - 監視・リファクタリング計画
- [SESSION_PROGRESS_EXTENDED.md](SESSION_PROGRESS_EXTENDED.md) - 統計記録用途

**具体例**:
```
After Session 27:

✅ SESSION_PROGRESS.md: 63行（目安 200行以内）
✅ APP_CANDIDATES.md: 172行（目安 200-300行）
👁️ agents.md: 422行（目安 150-200行 - 監視対象）
✅ Others: すべて目安内

Action: Session 28 で agents.md の簡潔版参照に変更
       → agents.md 本体は「詳細参考版」へ降格
```

## 📋 Tier 体系（クイックリファレンス）

| Tier | 用途 | 行数目安 | 更新頻度 | 判定基準 |
|------|------|--------|--------|---------|
| **1** | コアガイド | 100-150 | 最小限 | 最上位・常に参照 |
| **2** | 運用ガイド | 150-200 | 各セッション | 操作手順・トラブル対応 |
| **3** | ロードマップ | 200-300 | 各フェーズ | 計画・要件定義 |
| **4** | レジストリ | 200+ | 各セッション | 記録・統計管理 |
| **5** | 知識・理論 | 250-600 | 随時 | 深掘り・背景 |
| **6** | プロジェクト別 | 100-150 | プロジェクト完了時 | プロジェクト専有 |
| **7** | 補助・チェック | 200-250 | 随時 | 管理・品質ゲート |

詳細は [DOCUMENTATION_STRATEGY.md#-ドキュメント体系](DOCUMENTATION_STRATEGY.md#-ドキュメント体系) を参照
