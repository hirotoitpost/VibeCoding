## Vibe Coding メタ学習

### Vibe Coding とは？

> **「自然言語による指示から、AI が実装を完遂する開発パラダイム」**

Vibe Coding の核心：
- ユーザーが「何をしたいか」を明確に述べる
- AI エージェントが「どのように実装するか」を自動判断
- 単なる「翻訳」ではなく、**知識獲得と段階的改善のサイクル**

### 「Vibe」の正体

Vibe（ヴァイブ）= **感覚的理解**

Vibe Coding では：
- AI が「ここまでできるなら、こういう実装が妥当」という *感覚* を研ぎ澄ます
- ユーザーは「こういう感覚的要求」を明確に表現する力を磨く

### 実装過程で生じたエラーと学習

#### エラー 1: URL エンコーディング失敗

```python
# ❌ アンチパターン（最初の AI 実装）
url = f"https://api.openweathermap.org/data/2.5/weather?q={city}&appid={api_key}"

# 入力: "New York" → "New York" がそのまま渡される
# 結果: URLControl character error
```

**ユーザー指示：**
```
「'New York' で実行するとエラーになっています。
 URLエンコーディングの処理を確認してください」
```

**AI の修正**:
```python
# ✅ ベストプラクティス
from urllib.parse import urlencode

params = {'q': city, 'appid': api_key}
url = f"https://api.openweathermap.org/data/2.5/weather?{urlencode(params)}"

# 入力: "New York" → URL 自動エンコーディング → "New%20York"
# 結果: ✅ 成功
```

#### 学習ポイント

| 項目 | 学習内容 |
|------|--------|
| **関数選択** | 「URLパラメータ」= `urlencode()` を使う（文字列連結ではなく） |
| **エラー検出** | ユーザーが「此の入力パターンで失敗」と指摘 → AI が全体戦略を再考 |
| **修正方向** | 「パラメータの渡し方」ではなく「パラメータ構築の流儀」を変える |

---

#### エラー 2: テスト環境の隔離不足

```python
# ❌ アンチパターン（最初のテスト）
def test_fetch_with_cache():
    # 前提: キャッシュファイルが /tmp/weather_cache.json に存在
    result = fetch_weather("Tokyo", use_cache=True)
    assert result["name"] == "Tokyo"

# 問題：テスト 1 で キャッシュが作られる
#       → テスト 2 で その キャッシュが入り込む
#       → 結果: 環境依存テストになる
```

**ユーザー指示：**
```
「3つのテストが一度に失敗することがあります。
 テスト環境のクリーンアップをしてください」
```

**AI の修正**:
```python
# ✅ ベストプラクティス
def tearDown(self):
    """各テスト終了時にキャッシュファイルを削除"""
    cache_file = '/tmp/weather_cache.json'
    if os.path.exists(cache_file):
        os.remove(cache_file)

def test_fetch_with_cache(self):
    # use_cache=False で外部キャッシュに干渉しない
    result = fetch_weather("Tokyo", use_cache=False)
    assert result["name"] == "Tokyo"
```

#### 学習ポイント

| 項目 | 学習内容 |
|------|--------|
| **テスト独立性** | 各テストは `tearDown()` で後始末 → 他のテストへの影響 0 |
| **パラメータ制御** | `use_cache=False` で外部依存を明示的に排除 |
| **フレームワーク活用** | unittest の `tearDown()` は「後処理の標準イディオム」 |

## パターン集

### パターン A: API 連携（URL エンコーディング）

**問題**: スペースを含む検索キーワード（"New York"）をそのまま URL に埋込むと、制御文字エラーになる

**解決策**:
```python
# ✅ 正しい方法: urllib.parse.urlencode を使用
from urllib.parse import urlencode

params = {'q': city, 'appid': api_key}
url = base_url + '?' + urlencode(params)

# ❌ やってはいけない方法: 文字列連結
# url = f"{base_url}?q={city}&appid={api_key}"  # "New York" がそのまま
```

**Vibe Coding 的指示**:
```
「マルチワードのキーワード（例: 'New York'）を
 API に渡すときは、URL エンコーディングを
 ライブラリ関数で行ってください」
```

**AI の判断**:
- "URL パラメータ" = 標準ライブラリの `urlencode()`
- 自前でやらない（よくあるバグの源）

---

### パターン B: テスト隔離（tearDown）

**問題**: 前のテストで作られたキャッシュが、次のテストに混入する

**解決策**:
```python
# ✅ 正しい方法: tearDown で毎回 cleanup
def tearDown(self):
    if os.path.exists(self.cache_file):
        os.remove(self.cache_file)

# ❌ やってはいけない方法: setUp だけで初期化
def setUp(self):
    self.cache_file = '/tmp/weather_cache.json'
    # cleanup なし → テスト順序に依存
```

**Vibe Coding 的指示**:
```
「各テスト関数は独立していること。
 前のテストの副作用（キャッシュファイル）を
 後のテストが引き継がないように tearDown で
 クリーンアップしてください」
```

**AI の判断**:
- "テスト独立性" = unittest フレームワークの `tearDown()` メソッド
- 自動化手段あり（手動削除ではなく）

---

### パターン C: 依存関係の完全性

**問題**: `requirements.txt` に `pytest` がないので、`pip install` 後に pytest が使えない

**解決策**:
```
# ✅ 正しい方法: 全ての deps を記載
requests==2.31.0
python-dotenv==1.0.0
pytest==7.4.3  # ← テストツールも deps に含める

# ❌ やってはいけない方法: 「実行時 deps だけ」
requests==2.31.0
python-dotenv==1.0.0
# pytest は「だけ開発者が手動で pip install」→ new contributor はハマる
```

**Vibe Coding 的指示**:
```
「requirements.txt に、テストツール（pytest）も含めてください。
 新しい開発者が 'pip install -r requirements.txt' だけで
 テストまで走らせられるようにしてください」
```

**AI の判断**:
- "完全なセットアップ" = 実行・開発・テストの全 deps を記載

---

### パターン D: エラーハンドリングの階層化

**問題**: ネットワークエラー、API エラー、ローカル I/O エラーを区別できない

**解決策**:
```python
# ✅ 正しい方法: 例外の種類で分岐
try:
    response = requests.get(url)
except requests.exceptions.HTTPError as e:
    # API エラー (4xx, 5xx)
    if response.status_code == 401:
        print("エラー: API キー無効")
    elif response.status_code == 404:
        print("エラー: 都市が見つかりません")
except requests.exceptions.URLError as e:
    # ネットワークエラー
    print("エラー: ネットワーク接続できません")
except IOError as e:
    # ローカルファイル I/O エラー
    print("エラー: キャッシュファイルが読み込めません")

# ❌ やってはいけない方法: 全部「except Exception」で一括
except Exception as e:
    print("エラーが起きました")  # ← 何が起きたか分からない
```

**Vibe Coding 的指示**:
```
「以下のエラーシナリオを個別にハンドル:
 1. API キー無効 (401)
 2. 都市が見つからない (404)
 3. ネットワーク接続失敗
 4. キャッシュファイルの読み書き失敗
 各シナリオで異なるメッセージを出力してください」
```

**AI の判断**:
- "エラーの種類" = 例外クラスの使い分け
- 単一の catch-all ではなく、段階的な except 句

## AI エージェント協働のコツ

### 1. AI に「試行錯誤」を任せる

```
ユーザーからの指示:
「以下を試してみてください:
 1. キャッシュ有効期限を 5 分に設定
 2. 1 回目の呼び出しを計測
 3. 2 回目の呼び出しを計測
 4. 2 回目が 1 回目の 90% 以上高速か確認」
```

**勘所**: ユーザーが結果の検証方法を示すと、AI は試行錯誤できる

### 2. 「何をしたか」の履歴を残す

```
実装ログの例:

Step 1: urllib.parse をインポート
Step 2: params 辞書를 構築
Step 3: urlencode() で エンコード
Step 4: 最終 URL を構築
Step 5: テストで "New York" 入力で確認

→ AI がやったことが追跡可能
```

**勘所**: 完全なコードだけでなく、「何を考えてそうしたのか」が分かると、次の人が改良しやすい

### 3. 「完了条件の再確認」をメインのワークフローに入れる

```
AI の完了チェック:

[ ] 機能が 完了
[ ] テストが 成功
[ ] ドキュメントが 書かれた
[ ] ユーザーに 検証してもらった ← 重要！
```

**勘所**: ユーザーの検証がないと、「AI の想像」の可能性

## まとめ

### Vibe Coding の本質

Vibe Coding は「**指示の品質**」と「**AI の判断力**」の共進化プロセス。

1. **ユーザーは**: 曖昧さを減らし、要件を明確にする技術を磨く
2. **AI は**: 要件から最適な実装パターンを認識する能力を磨く
3. **結果**: 両者が協力すると、個別では到達できない品質に達する

### 実装者向けアクションアイテム

- [ ] 次のプロジェクト計画時に「指示設計の 5 段階」を使う
- [ ] エラー発生時に「最小再現例 (MRE)」を心がける
