## 開発・デバッグプロセス

### 🔧 トラブルシューティング・デバッグワークフロー

実装中に問題が発生した場合の標準的な対応プロセス：

#### Step 1: エラーメッセージの読み込みと分類
```
発生した症状を記録:
- エラーメッセージの正確なテキスト
- スタックトレース（最後の行が重要）
- 実行したコマンド
- 入力値・パラメータ
- 環境情報（OS、バージョン、ライブラリ）
```

**エラーの分類**:
- URLエラー（入力値の形式）
- インポート・依存関係エラー（環境）
- ロジックエラー（実装）
- テスト環境エラー（独立性）

#### Step 2: 原因の仮説立て
```
推定される原因:
- コード内の具体的な箇所
- 関連ライブラリのドキュメント確認
- 同様の問題の先例検索
- 外部 API やシステムの動作確認
```

#### Step 3: 問題の再現
```
最小限の再現例を作成:
- 単純なテストケース
- print やログでの値の追跡
- デバッグモードでの実行
```

#### Step 4: 原因の確定と修正
```
修正実装:
- 根本原因に対する修正
- 関連する他の箇所への波及確認
- エッジケース（複数単語、特殊文字など）への対応
```

#### Step 5: テスト・検証
```
修正の検証:
- 修正前後での動作比較
- 関連テストの実行
- エッジケースのテスト
- 本番環境での動作確認
```

#### Step 6: 学習と記録
```
今後への活用:
- 具体例をトラブルシューティングガイドに記録
  → examples/[プロジェクト]/TROUBLESHOOTING.md
- ベストプラクティスを docs/ に記録
- git commit で履歴に残す
```

## Git/GitHub ワークフロー

### 🔀 ブランチ戦略

#### ブランチ名の形式
```
feature/作業ID_主な作業内容のタイトル

例:
  feature/001_プロジェクト初期化
  feature/002_基礎プロジェクト実装
  feature/003_ツール調査ドキュメント
```

**作業IDについて**:
- 本来は Redmine チケット番号を使用
- 現状では AI エージェントが連番で発行（001, 002, 003...）
- 全作業で一意の ID を付与

#### ブランチ作成手順
```bash
# デフォルトブランチを最新に
git fetch origin
git checkout main
git pull origin main

# 作業ブランチをmainから生成
git checkout -b feature/作業ID_タイトル
```

---

### 💬 コミットメッセージの書式

#### テンプレート
```
refs #作業ID コミット内容のサマリー

詳細説明（複数行可）
- 変更点1
- 変更点2
```

#### 記入ルール
- 1行目: `refs #作業ID` + スペース + サマリー（50文字程度）
- 2行目: 空行
- 3行目以降: 詳細（何を、なぜ、どう変わったか）
- リンク: GitHub Issue/PR へ自動リンク

#### 例
```
refs #002 天気情報取得ツールの実装

- OpenWeatherMap API との連携
- エラーハンドリングの追加
- 5分キャッシング機能を実装
```

### 📤 プルリクエスト（PR）の投稿

#### PR 投稿タイミング
- **区切りの良いタイミング**: 各フェーズ完了時、大きな機能単位
- **推奨**: 1 PR = 1 論理的単位（複数ファイルの更新は OK）

#### PR タイトルの書式
```
refs #作業ID マージする作業内容のサマリー

例:
  refs #002 基礎プロジェクト（天気情報ツール）実装完了
```

#### PR 本文の書式
```
refs #作業ID マージする作業内容のサマリー

## 概要
この PR では[作業内容]を実装しました。

## 含まれる変更
[コミットID] コミットタイトル
[コミットID] コミットタイトル

例:
[a1b2c3d] refs #002 天気情報取得ツールの実装
[d4e5f6g] refs #002 テストケースの追加
[h7i8j9k] refs #002 README.md にセットアップ手順を追記

## レビューポイント
- [確認事項1]
- [確認事項2]

## テスト
- [ ] ローカルテスト成功
- [ ] エラーケーステスト成功
```

#### PR 投稿手順
```bash
# リモートに作業ブランチをプッシュ
git push origin feature/作業ID_タイトル

# GitHub の Web UI で PR を作成
# - Base: main
# - Compare: feature/作業ID_タイトル
# - タイトル・本文を記入
# - "Create Pull Request" をクリック
```

---

### ✅ PR マージ後の作業

#### ステップ 1: ユーザーがマージを宣言
ユーザーから「PR マージ完了」の通知を受ける

#### ステップ 2: フェッチとリベース
```bash
# リモートを最新に
git fetch origin

# main ブランチを最新に
git checkout main
git pull origin main

# 作業ブランチへ戻る
git checkout feature/作業ID_タイトル

# main に対してリベース
git rebase main
# 競合があれば解決して
# git add .
# git rebase --continue
```

#### ステップ 3: リモートブランチ削除後の処理
ユーザーがリモートブランチを削除した後：

```bash
# ローカルブランチを最新に
git fetch origin

# main を最新に
git checkout main
git pull origin main

# 作業ブランチを削除（ファストフォワード完了のため不要に）
git branch -d feature/作業ID_タイトル

# 次の作業へ
# 新しいブランチを作成して作業継続
```

## マージ競合解消プロセス

### 概要

複数の PR が同じファイルを修正する場合、マージ時に競合が発生します。

**ケーススタディ**: Session 4 で PR #1 と PR #2 が agents.md で競合した事例から、解消プロセスを標準化しました。

### 競合が発生する理由

```
master ブランチ状態（最初）
├─ agents.md （作業ID テーブル: ID 001-002）

PR #1: feature/005 → master
├─ agents.md を更新（ID 003-006 を追加）
└─ マージ完了 ✅

PR #2: feature/007 → master（PR #1 マージ前に作成）
├─ agents.md を更新（ID 003-008 を追加）
└─ マージ時に競合 ⚠️

原因: PR #2 は PR #1 マージ前のベースから作成されたため、
     同じ行（作業ID テーブル）を異なる内容で修正している
```

### 競合解消の手順

#### Step 1: リモートをフェッチして状態確認

```bash
git fetch origin
git checkout feature/007_指示設計ワークショップ
```

**確認項目**:
- リモートに最新の PR 情報が反映されているか
- ローカルブランチが最新か

#### Step 2: マージまたはリベースで競合を可視化

**方法 A: マージの場合**
```bash
git merge origin/master --no-commit
git status  # 競合ファイルを確認
```

**方法 B: リベースの場合（推奨 - 線形履歴）**
```bash
git rebase origin/master
# 競合が表示される
```

**出力例**:
```
CONFLICT (content): Merge conflict in agents.md
Rebasing (1/1)
error: could not apply 6bc9f9a...
```

#### Step 3: 競合マーカーを確認

競合ファイルを開くと、以下の形式で競合部分が表示されます:

```
<<<<<<< HEAD
（マージ先の内容 - origin/master）

=======
（マージ元の内容 - feature/007）

>>>>>>> コミットハッシュ
```

**agents.md の競合例**:
```
<<<<<<< HEAD
| 003 | Git/GitHub ワークフロー追加 | ⏳進行中 | TBD | セッション2 (続) |
| 004 | フェーズ2 理論学習 | ✅完了 | a31f06f... | セッション2 完了 |
=======
| 003 | Git/GitHub ワークフロー追加 | ✅完了 | 33c1173 | セッション2 (続) |
| 004 | フェーズ2 理論学習 | ✅完了 | TBD | セッション3 |
| 005 | フェーズ3.1 基礎プロジェクト... | ✅完了 | 7980dfb | セッション3 |
| 006 | トラブルシューティング文書化... | ✅完了 | 86d2146 | セッション3 |
| 007 | 指示設計ワークショップ... | ⏳進行中 | TBD | セッション4 |
>>>>>>> 6bc9f9a
```

#### Step 4: 競合を手動解決

**戦略**: 両方の情報を統合する

```markdown
解決後（統合版）:
| 003 | Git/GitHub ワークフロー追加 | ✅完了 | 33c1173 | セッション2 (続) |
| 004 | フェーズ2 理論学習 | ✅完了 | TBD | セッション3 |
| 005 | フェーズ3.1 基礎プロジェクト... | ✅完了 | 7980dfb | セッション3 |
| 006 | トラブルシューティング文書化... | ✅完了 | 86d2146 | セッション3 |
| 007 | 指示設計ワークショップ... | ✅完了 | 6bc9f9a | セッション4 |
| 008 | フェーズ3.2 中級プロジェクト | ⏸待機 | TBD | 次セッション予定 |
```

**重要**: 競合マーカー（`<<<<<<<`, `=======`, `>>>>>>>`）を完全に削除する

#### Step 5: ファイルを追加して競合解決をコミット

**マージの場合**:
```bash
git add agents.md
git commit -m "Merge branch 'feature/007' into master"
```

**リベースの場合**:
```bash
git add agents.md
git rebase --continue
```

**出力例**:
```
[detached HEAD 85465d8] refs #007 Vibe Coding 指示設計パターン・ガイド完成
 3 files changed, 946 insertions(+), 9 deletions(+)
Successfully rebased and updated refs/heads/feature/007_指示設計ワークショップ.
```

#### Step 6: リモートに反映

```bash
git push origin feature/007 --force-with-lease
```

**フラグ説明**:
- `--force-with-lease`: 安全な強制プッシュ（リモートに新しい変更がないことを確認）
- より安全な `--force` よりも推奨

**出力例**:
```
+ 6bc9f9a...85465d8 feature/007_指示設計ワークショップ (forced update)
```

#### Step 7: GitHub で PR を確認

リモートプッシュ後、GitHub の PR ページが自動的に更新されます:
- 競合マークが消える
- 「Merge pull request」ボタンが活性化

### トラブルシューティング

#### 問題: 競合解決中にミスした

**解決方法**:
```bash
git merge --abort           # マージ中の場合
# または
git rebase --abort          # リベース中の場合

# 最初から始める
git merge origin/master --no-commit  # または git rebase origin/master
```

#### 問題: どちらの変更を採用すべきか判断できない

**手順**:
1. 両方のファイル内容を確認: `git show HEAD:agents.md` と `git show <branch>:agents.md`
2. 差分を表示: `git diff origin/master agents.md`
3. チームと相談して方針を決める
4. 統合版を作成

#### 問題: `--force-with-lease` でもプッシュが失敗した

**原因**: リモートに別の更新がある

**確認と解決**:
```bash
git fetch origin
git log --oneline origin/feature/007..HEAD
# コミット差分がなければ、既にマージされている可能性あり
```

### ベストプラクティス

#### ✅ 推奨

1. **複数 PR は依存関係を明示する**
   ```
   PR #2 の説明に「Depends on PR #1」と記載
   ```

2. **PR 作成時はマージ予定順を決める**
   ```
   PR #1（ID 005-006） → 先にマージ
   PR #2（ID 007-008） → 後にマージ
   ```

3. **競合検出時は早期にコミュニケーション**
   ```
   PR コメントで「競合を検出しました、リベースで解決します」と通知
   ```

4. **リベースよりマージを使う場合のルール**
   ```
   連続した複数 PR の場合でも、各 PR のマージコミットで
   「どの PR がマージされたか」を明確に記録する
   ```

#### ❌ 避けるべき

1. **`git push --force`（flagなし）を使う** → `--force-with-lease` を使う
2. **競合を無視して重要な行を削除する** → 全行を確認
3. **競合解決を AI に全委任** → 人間が最終確認
4. **マージ戦略を統一しない** → チームで決めておく

### 学習ポイント（Session 4）

| 項目 | 学習内容 | 応用 |
|------|--------|------|
| **リベース vs マージ** | リベースで線形履歴、マージでマージコミット記録 | Phase 3.2+ で戦略を統一 |
| **競合マーカー** | `<<<<<<<` など 7 文字マーカーは完全に削除 | テーブルやコード行では特に注意 |
| **force-with-lease** | リモートの新規更新を防ぐ安全な強制プッシュ | 複数人で同時開発するときは必須 |
| **競合の根本原因** | PR 作成時点のベースが異なると競合発生 | 依存関係を事前に宣言して計画 |
