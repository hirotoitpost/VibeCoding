# Example 01: コードハイライト

## キャッシング実装
```python
cache = {}
CACHE_TTL = 300  # 5分

def get_weather(city):
  if cache_valid(city):
    return cache[city]
  data = api_call(city)
  cache[city] = data
  return data
```

## エラーハンドリング
```python
try:
  response = requests.get(...)
except requests.ConnectionError:
  print("ネットワークエラー")
except KeyError:
  print("都市が見つかりません")
```

<!-- 
ナレーター:
「実装のハイライトをお見せします。
まずキャッシング機構。cache という辞書を用意して、
city をキーに、データをバリューで保存します。
CACHE_TTL で有効期限を 300 秒（5分）に設定。
get_weather 関数内で、まずキャッシュを確認して、
有効なら即座に返す。ないか期限切れなら API を呼び出すというロジックです。

続いてエラーハンドリング。
try-except で複数のエラーをキャッチしています。
実務では こういった細かな対応が必須です。」

# Example 02: 3層アーキテクチャ

```
┌─────────────────────────┐
│  フロントエンド (React) │
│  - 入力フォーム          │
│  - 一覧表示              │
│  - グラフ表示            │
└─────────────┬───────────┘
      HTTP(JSON)
┌─────────────↓───────────┐
│ バックエンド (Express)  │
│  - POST /api/items      │
│  - GET /api/items       │
│  - PUT /api/items/:id   │
│  - DELETE /api/items/:id│
└─────────────┬───────────┘
        SQL
┌─────────────↓───────────┐
│DB (SQLite/PostgreSQL)   │
│  - items テーブル        │
└─────────────────────────┘
```

<!-- 
ナレーター:
「では、3層アーキテクチャを見てみましょう。
最上層は React のフロントエンド。ユーザーは ここで収支を入力したり、
グラフで月ごとの推移を見たり、履歴を管理したりします。
中間層が Express のバックエンド。フロント からのリクエスト（JSON）を受け取って、
データベースに対して CRUD 操作を実行。結果を JSON で返します。
最下層はデータベース。SQLite や PostgreSQL を使って、収支データを永続化します。」

アシスタント:
「つまり、ユーザーがフロントで『追加』をクリックすると、
フロント→バック→DB という流れでデータが保存される…と。」

ナレーター:
「その通り。そして逆に『一覧を表示してくれ』というクエリも同じルーを通ります。
この3層分離により、保守性と拡張性が大幅に向上します。」

# Example 03: MQTT & Pub/Sub

```
センサー① (温度)    センサー② (湿度)
     ↓                    ↓
┌──────────────────────────┐
│   MQTT ブローカー         │
│   (Mosquitto)            │
│   topics:                │
│   ├─ /sensors/temp       │
│   ├─ /sensors/humidity   │
│   └─ /sensors/pressure   │
└──────────────────────────┘
            ↑↓
  Web ダッシュボード
  (Node.js + WebSocket)
  リアルタイム更新
```

## 💡 メリット
✅ スケーラブル＆疎結合
✅ リアルタイム性能
✅ センサー独立動作

<!-- 
ナレーター:
「MQTT は Pub/Sub（パブ・サブ）というパターンです。
複数のセンサーが Publisher の役割で、

アシスタント:

# Example 06: Solidity ERC-20

```solidity
pragma solidity ^0.8.20;

contract VibeCodingToken is
    ERC20("VibeCoding", "VBC")
{
  // Mint: 新規トークン発行
  function mint(uint256 amount)
    public { ... }
  
  // Transfer: 送信
  function transfer(addr, amt)
    public { ... }
  
  // Approve: 自動脱出許可
  function approve(addr, amt)
    public { ... }
}
```

## ✨ ERC-20 標準仕様
- トークン名・シンボル管理
- 残高管理
- 送受信（transfer）
- 承認・委任（approve）
- イベント emit

<!-- 
ナレーター:
「では、Solidity コードを見てみましょう。
『VibeCodingToken』という ERC-20 準拠のトークンを定義しています。
ERC-20 とは、イーサリアム上の標準的なトークン仕様です。
重要な機能は 3 つ。
まず Mint（鋳造）。新規トークンを発行する関数です。
次に Transfer（送信）。トークン保有者が他のアドレスに送ります。
そして Approve（承認）。保有者が第三者に一定額の使用を許可する…
これが『委任』という概念です。」

# Example 06: Hardhat 開発フロー

```
1️⃣ コード実装 (Solidity)
      ↓
2️⃣ コンパイル ($ hardhat compile)
   ✅ Bytecode 生成
   ✅ ABI 生成
      ↓
3️⃣ テスト ($ hardhat test)
   ✅ 14/14 テスト PASS
   ✅ ガス推定
      ↓
4️⃣ Testnet デプロイ (Sepolia)
   ($ hardhat run scripts/deploy.ts)
   ✅ Contract アドレス取得
      ↓
5️⃣ Testnet で動作確認
   (Etherscan で TX確認)
   ✅ TX ハッシュ確認
   ✅ ブロック確認
```

<!-- 
ナレーター:
「Hardhat での開発フローです。
