## 📋 PR メッセージテンプレート

### 基本フォーマット

```markdown
## 変更の概要
[PR で実装した機能・修正内容の要約。60-80 文字で簡潔に]

## 変更の詳細
[何を変更したのか、なぜ変更したのかを記載]

- 項目 1: [変更内容]
- 項目 2: [変更内容]

## セッション・Work ID
- **Session**: XX
- **ID**: XXX
- **フェーズ**: X.X

## テスト実施
- [ ] ローカルテスト実施済み
- [ ] ドキュメント確認済み
- [ ] 環境変数確認済み

## チェックリスト
- [ ] コミットメッセージが規約に準拠
- [ ] ドキュメントが更新済み
- [ ] 不要なファイルが含まれていない
- [ ] .gitignore に追加すべきファイルがある場合は追加済み

## 関連 Issue / PR
- Closes #123
- Related-To: ID XXX, ID YYY
```

## ✅ PR 作成チェックリスト

### 前チェック（Push 前）

- ✅ **ブランチ名が正しい**
  - `feature/[ID]_[タイトル]` (新機能)
  - `fix/[ID]_[説明]` (バグ修正)
  - `docs/session-[X]-update` (ドキュメント)
  
- ✅ **コミットが規約に準拠**
  - [GIT_COMMIT_CONVENTION.md](GIT_COMMIT_CONVENTION.md) 確認
  - 最新コミット: `git log --oneline -1`

- ✅ **ローカルテスト**
  - 期待される出力が得られたか
  - エラーハンドリングが機能しているか
  - ドキュメントが最新か

### PR 作成時

- ✅ **タイトルは明確**
  - `feat(ID 027): AviUtl Exo ファイル INI 形式書き直し`
  - `docs(Session 27): ドキュメント整理完了`

- ✅ **説明文が充実**
  - What（何を実装したか）
  - Why（なぜ必要か）
  - How（どのように実装したか）

- ✅ **ファイル差分が最小限**
  - 不要な変更がないか確認
  - `.env` や `__pycache__/` が含まれていないか確認

### PR レビュー後

- ✅ **レビューコメントへの対応**
  - 指摘があれば、新規コミットで修正
  - `git commit --amend` で修正履歴をクリーンに
  - Force push は避ける（競合の原因）

- ✅ **マージ前確認**
  - すべてのコンフリクトが解消されたか
  - CI / 自動テストが通ったか（あれば）
  - ドキュメント更新が反映されたか

## 🔴 よくある间違い

### 1. PR タイトルが曖昧
```
❌ "Update"
❌ "Fix bugs"
❌ "WIP: 作業中"

✅ "feat(ID 027): Exo ファイル INI 形式書き直し"
✅ "fix(generate_exo): テキスト形式判定エラー"
```

### 2. ファイルが多すぎる
```
❌ 30 ファイル以上の変更
❌ 無関係なファイルが混在
❌ `.env`, `output/` が含まれている

✅ 関連ファイルのみ
✅ .gitignore で除外されたファイルのみ
```

### 3. コミットメッセージが乱雑
```
❌ "aaa"
❌ "temp fix"
❌ "ここまで部分修正"

✅ [GIT_COMMIT_CONVENTION.md](GIT_COMMIT_CONVENTION.md) に準拠
✅ 何を変更したか、なぜか明確
```

### 4. ドキュメント未更新
```
❌ PR: コード実装のみ
❌ ドキュメントと実装が不一致

✅ コード実装 + ドキュメント同時更新
✅ SESSION_PROGRESS.md / WORK_ID_REGISTRY.md 更新
```

## 📊 PR タイプ別テンプレート

### 機能実装 (feat)

```markdown
## 変更の概要
[新機能の簡潔な説明]

## 実装内容
- [ ] バックエンド: [ファイル名] - [説明]
- [ ] フロントエンド: [ファイル名] - [説明]
- [ ] テスト: [テスト内容]

## セッション・ID
- Session: XX
- ID: XXX
- Phase: X.X

## テスト確認
- [ ] ローカル実行確認済み
- [ ] エラーハンドリング確認済み
- [ ] ドキュメント更新済み
```

### バグ修正 (fix)

```markdown
## 変更の概要
[バグの説明と修正内容]

## 根本原因
[なぜバグが発生したのか]

## 修正内容
- ファイル: [ファイル名]
- 変更箇所: [行数・関数名]
- 修正方法: [何をしたか]

## テスト確認
- [ ] 修正前は エラーが再現したか
- [ ] 修正後は正常に動作するか
```

### ドキュメント更新 (docs)

```markdown
## 変更の概要
[ドキュメント更新内容]

## 更新ファイル
- [ ] SESSION_PROGRESS.md
- [ ] WORK_ID_REGISTRY.md
- [ ] [ドキュメント名]

## 変更理由
[ドキュメント更新が必要な理由]

## 行数変化
- Before: XX 行
- After: YY 行
```
