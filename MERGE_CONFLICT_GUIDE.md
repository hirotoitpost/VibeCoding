# マージ競合解消プロセス

## 概要

複数の PR が同じファイルを修正する場合、マージ時に競合が発生します。このドキュメントでは、Session 4 で実際に発生した競合事例を基に、標準化されたプロセスを記載します。

---

## ケーススタディ

### 競合が発生した理由

```
main ブランチ状態（初期）
├─ agents.md （作業ID テーブル: ID 001-002）

PR #1: feature/005 → main
├─ agents.md を更新（ID 003-006 を追加）
└─ マージ完了 ✅

PR #2: feature/007 → main（PR #1 マージ前に作成）
├─ agents.md を更新（ID 003-008 を追加）
└─ マージ時に競合 ⚠️

原因: PR #2 は PR #1 マージ前のベースから作成されたため、
     同じ行（作業ID テーブル）を異なる内容で修正している
```

---

## 競合解消の7ステップ

### Step 1: リモートをフェッチして状態確認

```bash
git fetch origin
git checkout feature/007_指示設計ワークショップ
```

**確認項目**:
- リモートに最新の PR 情報が反映されているか
- ローカルブランチが最新か

### Step 2: マージまたはリベースで競合を可視化

**方法 A: マージの場合**
```bash
git merge origin/main --no-commit
git status  # 競合ファイルを確認
```

**方法 B: リベースの場合（推奨 - 線形履歴）**
```bash
git rebase origin/main
# 競合が表示される
```

**出力例**:
```
Auto-merging agents.md
CONFLICT (content): Merge conflict in agents.md
Rebasing (1/2)
error: could not apply 6bc9f9a...
```

### Step 3: 競合マーカーを確認

競合ファイルを開くと、以下の形式で競合部分が表示されます:

```
<<<<<<< HEAD
（マージ先の内容 - origin/main）

=======
（マージ元の内容 - feature/007）

>>>>>>> コミットハッシュ
```

**実例（agents.md の作業ID テーブル）**:
```
<<<<<<< HEAD
| 003 | Git/GitHub ワークフロー追加 | ✅完了 | bf27c1f | セッション2 (続) |
| 004 | フェーズ2 理論学習 | ✅完了 | PR #3 (408856b) | セッション2 完了 |
| 005 | フェーズ3.1 基礎プロジェクト（天気情報ツール） | ✅完了 | PR #1 (b9bb217) | セッション3 |
| 006 | トラブルシューティング文書化・プロセス標準化 | ✅完了 | 86d2146 | セッション3 |
| 007 | 指示設計ワークショップ（Vibe Coding パターン体系化） | ⏳ PR #2 待機中 | PR #2 (未マージ) | セッション4 |
=======
| 003 | Git/GitHub ワークフロー追加 | ✅完了 | 33c1173 | セッション2 (続) |
| 004 | フェーズ2 理論学習 | ✅完了 | TBD | セッション3 |
| 005 | フェーズ3.1 基礎プロジェクト（天気情報ツール） | ✅完了 | 7980dfb | セッション3 |
| 006 | トラブルシューティング文書化・プロセス標準化 | ✅完了 | 86d2146 | セッション3 |
| 007 | 指示設計ワークショップ（Vibe Coding パターン体系化） | ✅完了 | 6bc9f9a | セッション4 |
>>>>>>> 6bc9f9a
```

### Step 4: 競合を手動解決

**戦略**: 両方の情報を統合し、最新の状態を反映

```markdown
解決後（統合版）:
| 003 | Git/GitHub ワークフロー追加 | ✅完了 | bf27c1f | セッション2 (続) |
| 004 | フェーズ2 理論学習 | ✅完了 | PR #3 (408856b) | セッション2 完了 → PR #3 マージ完了 |
| 005 | フェーズ3.1 基礎プロジェクト（天気情報ツール） | ✅完了 | PR #1 (b9bb217) | セッション3 → PR #1 マージ完了 |
| 006 | トラブルシューティング文書化・プロセス標準化 | ✅完了 | 86d2146 | セッション3 → PR #1 に含む |
| 007 | 指示設計ワークショップ（Vibe Coding パターン体系化） | ⏳ PR #2 待機中 | PR #2 (未マージ) | セッション4 → マージ待ち |
| 008 | フェーズ3.2 中級プロジェクト | ⏸待機 | TBD | 次セッション予定 |
```

**⚠️ 重要**: 競合マーカー（`<<<<<<<`, `=======`, `>>>>>>>`）を完全に削除する

### Step 5: ファイルを追加して競合解決をコミット

**マージの場合**:
```bash
git add agents.md
git commit -m "Merge branch 'feature/007' into master"
```

**リベースの場合**:
```bash
git add agents.md
git rebase --continue
```

**出力例**:
```
[detached HEAD 1a11438] refs #007 Vibe Coding 指示設計パターン・ガイド完成
 3 files changed, 937 insertions(+), 3 deletions(-)
Successfully rebased and updated refs/heads/feature/007_指示設計ワークショップ.
```

### Step 6: リモートに反映

```bash
git push origin feature/007_指示設計ワークショップ --force-with-lease
```

**フラグ説明**:
- `--force-with-lease`: 安全な強制プッシュ（リモートに新しい変更がないことを確認）
- より危険な `--force` よりも推奨

**出力例**:
```
+ c8fb070...16292c9 feature/007_指示設計ワークショップ (forced update)
```

### Step 7: GitHub で PR を確認

リモートプッシュ後、GitHub の PR ページが自動的に更新されます:
- ✅ 競合マークが消える
- ✅ 「Merge pull request」ボタンが活性化

## トラブルシューティング

### ❌ 問題: 競合解決中にミスした

**解決方法**:
```bash
git merge --abort           # マージ中の場合
# または
git rebase --abort          # リベース中の場合

# 最初から始める
git merge origin/master --no-commit  # または git rebase origin/master
```

### ❌ 問題: どちらの変更を採用すべきか判断できない

**手順**:
1. 両方のファイル内容を確認:
   ```bash
   git show HEAD:agents.md        # master 版
   git show feature/007:agents.md # feature 版
   ```
2. 差分を表示: `git diff origin/master agents.md`
3. チームと相談して方針を決める
4. 統合版を作成

### ❌ 問題: `--force-with-lease` でもプッシュが失敗した

**原因**: リモートに別の更新がある

**確認と解決**:
```bash
git fetch origin
git log --oneline origin/feature/007..HEAD
# コミット差分がなければ、既にマージされている可能性あり
```

## ベストプラクティス

### ✅ 推奨する方法

1. **複数 PR は依存関係を明示する**
   ```
   PR #2 の説明に「Depends on PR #1」と記載
   ```

2. **PR 作成時はマージ予定順を決める**
   ```
   PR #1（ID 005-006） → 先にマージ
   PR #2（ID 007-008） → 後にマージ
   ```

3. **競合検出時は早期にコミュニケーション**
   ```
   PR コメントで「競合を検出しました、リベースで解決します」と通知
   ```

4. **リベースよりマージを使う場合のルール**
   ```
   連続した複数 PR の場合でも、各 PR のマージコミットで
   「どの PR がマージされたか」を明確に記録する
   ```

### ❌ 避けるべき方法

1. **`git push --force`（フラグなし）を使う**
   → `--force-with-lease` を使う

2. **競合を無視して重要な行を削除する**
   → 全行を確認して統合

3. **競合解決を AI に全委任**
   → 人間が最終確認する

4. **マージ戦略を統一しない**
   → チームで決めておく

## 学習ポイント（Session 4）

| 項目 | 学習内容 | 応用シーン |
|------|--------|----------|
| **リベース vs マージ** | リベースで線形履歴、マージでマージコミット記録 | Phase 3.2+ で戦略を統一 |
| **競合マーカー** | `<>>>>` など 7 文字マーカーは完全に削除 | テーブルやコード行では特に注意 |
| **force-with-lease** | リモートの新規更新を防ぐ安全な強制プッシュ | 複数人で同時開発するときは必須 |
| **競合の根本原因** | PR 作成時点のベースが異なると競合発生 | 依存関係を事前に宣言して計画 |
